Главная задача

Удалить или деактивировать текущую самописную логику pagination / page breaking / paragraph splitting / line filling, которая сейчас вызывает нестабильную книжную верстку, и создать новый layout engine на основе лучших доступных библиотек и проверенных типографических подходов.

Текущая реализация больше не должна дорабатываться точечно.
Её нужно считать архитектурно неудачной для книжной верстки.

Почему текущую систему нужно заменить

Текущий движок ведёт себя как обычный web-layout, а не как профессиональный книжный typesetting engine.

Из-за этого возникают повторяющиеся ошибки:

страницы заканчиваются на разной высоте

внизу страниц остаётся слишком много пустого пространства

page break принимается слишком рано

появляются некрасивые разрывы абзацев

есть висячие строки и слабые хвосты

переносы слов выглядят рваными

страницы с заголовками считаются нестабильно

page number / footer логически смешан с flow

движок принимает локальные решения вместо глобальной оптимизации.

Поэтому задачу нужно решать не патчами, а заменой ядра верстки.

1. Что нужно сделать
1.1 Полностью убрать старую логику верстки

Удалить, отключить или изолировать текущие модули, отвечающие за:

line fitting

page filling

paragraph splitting

page break decision

footer collision avoidance

widow/orphan fixes

local overflow handling.

Запрещено поверх старой системы делать ещё один слой костылей.

1.2 Создать новый движок верстки книги

Новый движок должен строиться не как набор div-контейнеров, а как полноценная pipeline-система книжной верстки:

Book Content
→ Text Normalization
→ Paragraph Layout
→ Line Breaking
→ Hyphenation
→ Candidate Page Breaks
→ Page Scoring
→ Page Template Rendering
→ PDF Output
2. Архитектурный принцип

Moodra должен использовать готовые проверенные библиотеки, а не заново изобретать типографическое ядро.

Новый движок должен быть построен на основе:

основа страниц и paged media

Paged.js или Vivliostyle
(предпочтительно сначала проверить Paged.js как более простой для интеграции)

переносы слов

библиотека словарных переносов, например:

Hyphenation.js

Hypher

либо другой dictionary-based hyphenation engine

алгоритм качественного line breaking

реализация на основе:

Knuth–Plass

либо simplified paragraph optimization engine, если полный Knuth–Plass слишком тяжёлый на первом этапе.

3. Какой стек использовать
Предпочтительный стек
3.1 Page layout / paged media

Использовать Paged.js как базу для page model, page breaking и paged layout.

Нужно проверить, подходит ли он под:

A5 / custom trim size

page headers / footers

chapter start templates

export pipeline

Если Paged.js не покрывает нужный уровень контроля, использовать Vivliostyle.

3.2 Hyphenation

Использовать словарный движок переносов.

Переносы должны быть:

только по valid language breakpoints

всегда с дефисом

с лимитом по числу переносов на страницу

с penalty за перенос у нижней границы страницы.

3.3 Paragraph layout

Использовать paragraph optimization вместо greedy line fitting.

Если возможно — внедрить Knuth–Plass-style paragraph layout.
Если это слишком тяжело сразу — внедрить упрощённый paragraph scoring engine:

генерировать несколько вариантов line breaks

учитывать stretch/shrink пробелов

учитывать длину последней строки

учитывать hyphen penalties

выбирать лучший вариант.

4. Что новый движок обязан уметь
4.1 Страницы как шаблоны

Нужно ввести page templates:

NormalTextPage
ChapterStartPage
TechnicalPage
CopyrightPage
TOCPage

Их нельзя считать одинаковыми.

4.2 Footer должен быть вне flow

Номер страницы должен рендериться отдельно от основного текстового потока.

Страница должна иметь структуру:

Page
 ├ TopMargin
 ├ ContentArea
 ├ FooterSafeZone
 └ Footer

Текст не может входить в:

FooterSafeZone

Footer

4.3 Baseline grid

Все обычные текстовые страницы должны использовать единый vertical rhythm.

Нужно ввести:

baseline grid
preferred bottom baseline
target fill band

Обычные страницы должны заканчиваться примерно на одном уровне.

4.4 Page density balancing

Новый движок должен не просто “умещать текст”, а выравнивать плотность страниц.

Для normal text pages ввести целевой диапазон заполнения:

targetFillRatio = 92%–97%
4.5 Candidate breakpoints

Нельзя выбирать первый доступный page break.

Нужно тестировать несколько соседних breakpoints:

N
N+1
N+2
N+3
N+4

И выбирать лучший по score.

4.6 Page scoring

Каждый кандидат страницы должен получать score по совокупности правил.

Пример score:

unusedSpacePenalty
+ widowPenalty
+ orphanPenalty
+ uglyParagraphSplitPenalty
+ hyphenPenalty
+ shortLastLinePenalty
+ bottomRhythmPenalty
+ footerCollisionPenalty

Выбирается вариант с минимальным penalty.

5. Типографические правила
5.1 Widow / Orphan control

Нельзя оставлять:

одну строку абзаца внизу страницы

одну строку абзаца вверху следующей страницы.

5.2 Запрет слабых хвостов

Нельзя оставлять внизу страницы:

одно слово

слишком короткую строку

строку, визуально слишком слабую для завершения страницы.

5.3 Ограничение переносов

Нужно ввести:

maxHyphensPerPage = 3
maxConsecutiveHyphenLines = 2
5.4 Перенос слова только с дефисом

Если слово переносится между строками, дефис обязателен.

5.5 Переносы у края страницы штрафуются

Нужно сильно штрафовать переносы:

в последней строке страницы

в первых строках новой страницы

в местах, где они создают визуально рваный набор.

5.6 Продолжение абзаца на новой странице

Если абзац продолжается после page break:

первая строка продолжения не получает first-line indent

нет дополнительного spacing before.

6. Страницы начала главы

ChapterStartPage должна быть отдельным template.

Нельзя считать первую страницу главы как обычную страницу.

Нужно учитывать:

chapter label

title

decorative separator

gap after header

И только после измерения header block считать доступную текстовую область.

7. Технические страницы

Нужно выделить отдельный режим верстки для:

титульной страницы

содержания

копирайта

аннотации

посвящения.

Эти страницы не должны использовать те же правила page fill, что и обычный основной текст.

8. Экспорт PDF

PDF нельзя строить как случайный snapshot web DOM.

Нужно генерировать PDF из page model.

Каждая страница должна быть уже логически собрана как page object, а потом экспортирована.

9. Что нужно сделать по этапам
Этап 1 — отключение старого ядра

найти все модули текущего pagination engine

выключить старую логику page fill

выключить старые костыли вокруг widow/orphan

убрать старый footer-overlap fix.

Этап 2 — прототип нового ядра

интегрировать Paged.js или Vivliostyle

подключить hyphenation library

сделать прототип normal text page

сделать page model с footer safe zone.

Этап 3 — paragraph engine

заменить greedy line breaking

внедрить paragraph optimization

внедрить penalties

внедрить candidate breakpoints.

Этап 4 — page scoring

добавить target fill band

добавить bottom rhythm penalty

добавить unused space penalty

добавить bad split penalties.

Этап 5 — специальные шаблоны

chapter start page

toc

copyright

annotation

dedication.

Этап 6 — финальный PDF pipeline

page rendering

export

regression tests.

10. Что запрещено делать

Запрещено:

продолжать чинить старый движок локальными патчами

делать page break только по overflow

решать разную высоту страниц растяжением line-height

решать плотность страниц изменением paragraph spacing

вставлять искусственные пустые строки

смешивать footer с основным flow.

11. Что считать успешным результатом

Новый движок считается успешным, если:

страницы обычного текста заканчиваются примерно на одном уровне

footer никогда не конфликтует с текстом

нет обрезанных строк

нет рваных переносов

нет раннего page break без причины

chapter pages считаются корректно

абзацы продолжаются логично

книга визуально выглядит как профессионально сверстанная.

12. Минимальный MVP нового движка

Если невозможно сразу сделать всё, то сначала нужно реализовать MVP:

NormalTextPage

footer safe zone

dictionary hyphenation

paragraph optimization

candidate page breaks

page scoring

PDF export

Только после этого добавлять chapter templates и технические страницы.

13. Техническое требование к реализации

Не чинить текущую систему.
Пересобрать ядро как новый модуль, например:

BookLayoutEngineV2

Старый движок оставить только как временный fallback до миграции, но не развивать его.
