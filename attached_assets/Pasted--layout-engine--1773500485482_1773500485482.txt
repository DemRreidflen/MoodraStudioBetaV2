Мастер-промт для исправления оставшихся багов книжной верстки

Нужно доработать layout engine книжной верстки. Сейчас базовая пагинация уже работает лучше, но движок всё ещё нарушает профессиональные правила книжной типографики.

Нужно исправить следующие ошибки:

внизу страницы остаются слишком короткие хвосты строк, иногда буквально одно слово

последняя строка иногда визуально слишком близко к номеру страницы или залезает в его зону

переносы слов местами происходят без дефиса

переносов слов местами слишком много подряд

page break иногда происходит слишком рано, хотя место на странице ещё есть

движок иногда делает визуально слабый перенос полуслова или короткого хвоста в конце страницы.

Все исправления нужно делать на уровне layout engine и page-break rules, а не косметически через CSS.

1. Ввести strict bottom line quality control

Перед подтверждением page break движок обязан проверять качество последней строки на странице.

Нельзя оставлять внизу страницы:

строку из одного слова

строку с очень коротким хвостом

строку, ширина которой меньше визуально допустимого минимума

строку, которая слишком близко подходит к зоне footer / page number.

Правила:

minBottomLineWords = 2
minBottomLineWidth = 40% of text measure
minDistanceFromFooter = fixed safe gap

Если последняя строка не проходит хотя бы одно из правил, page break в этом месте запрещён.

2. Запретить строку, которая почти касается номера страницы

Сейчас на некоторых страницах нижняя строка не обрезана формально, но визуально слишком близка к номеру страницы и выглядит грязно.

Нужно ввести footer safe zone.

Структура страницы:

Page
 ├ Header
 ├ ContentArea
 ├ FooterSafeZone
 └ Footer

Где:

Footer содержит номер страницы

FooterSafeZone — фиксированная защитная область над номером страницы

текст не может заходить ни в Footer, ни в FooterSafeZone.

Правило:

lastLineBottom <= contentAreaBottom - footerSafeGap

Нельзя разрешать строку, если она только “формально влазит”.
Она должна влазить типографически комфортно.

3. Исправить перенос слов: дефис обязателен

Если слово переносится между строками, перенос должен всегда обозначаться дефисом.

Нельзя:

разрывать слово без дефиса

переносить часть слова на следующую строку молча

показывать сломанное слово как будто это новый фрагмент текста.

Правило:

if wordIsSplitAcrossLines:
    renderHyphenAtLineEnd = true

Если движок не может корректно вставить дефис — такой перенос запрещён.

4. Ограничить количество переносов слов

Сейчас по скринам видно, что местами переносов слишком много, и страница начинает выглядеть грязно.

Нужно добавить жёсткие лимиты:

maxHyphensPerPage = 3
maxConsecutiveHyphenatedLines = 2

Если лимит превышается, движок должен:

попробовать другой вариант line breaks

переразбить абзац

попробовать page break в другом месте

только в крайнем случае оставить перенос.

Переносы — это не норма по умолчанию, а запасной инструмент.

5. Переносы слов должны использовать словарь языка

Все переносы должны идти только по корректным hyphenation points.

Нельзя:

разрывать слово в случайном месте

разрывать слово по визуальному остатку, если словарь не подтверждает это место.

Использовать hyphenation dictionary для языка книги.

Пример логики:

hyphenationPoints = getValidHyphenationPoints(word, language)
choose latest valid point that fits

Если корректной точки нет — слово переносить нельзя.

6. Page break нельзя делать слишком рано

Сейчас на части страниц видно пустое пространство, хотя вниз ещё можно было положить строку или кусок абзаца.

Значит движок иногда слишком рано подтверждает перенос страницы.

Нужно изменить правило:

Нельзя делать page break только потому, что следующий фрагмент абзаца “не идеален целиком”.

Сначала движок обязан проверить:

можно ли разместить ещё одну строку

можно ли переразбить текущий абзац

можно ли использовать другой line break variant

можно ли убрать лишний перенос слова и тем самым уплотнить абзац.

Только после этого можно делать page break.

7. Запретить ugly paragraph split

Разрыв абзаца между страницами запрещён, если:

внизу страницы остаётся слишком короткий хвост

наверху следующей страницы остаётся слишком короткое продолжение

перенос выглядит как полуслово в плохом месте

абзац можно было лучше перераспределить.

Правило:

noUglyParagraphSplit = true
8. Добавить paragraph balancing перед каждым page break

Перед тем как подтвердить page break внутри абзаца, движок должен пересобрать абзац в нескольких вариантах.

Алгоритм:

paragraph
→ generate multiple legal line-break variants
→ evaluate each variant
→ choose variant with lowest badness score

Штрафы в badness score:

single-word bottom line

too short bottom line

too many hyphens

missing hyphen on split

line too close to footer

early page break with unnecessary whitespace

ugly paragraph split.

9. Ввести penalties для лишнего пустого пространства

Если движок делает page break и оставляет заметную пустую область, это должно штрафоваться.

Пример:

earlyPageBreakPenalty = high
unusedBottomSpacePenalty = proportional

То есть чем больше пустого места осталось без необходимости, тем хуже вариант layout.

Но это правило подчиняется более важным правилам:

no clipped lines

no footer collision

no ugly bottom runt

no bad split.

10. Перенос слова у нижней границы страницы должен быть особенно строгим

Если слово переносится в последней строке страницы, нужно проверять это жёстче, чем в середине страницы.

Запретить перенос внизу страницы, если одновременно выполняется хотя бы одно условие:

перенос создаёт слишком короткий хвост строки

перенос слишком близко к footer safe zone

перенос становится одним из нескольких подряд

на следующей странице продолжение слова выглядит как некрасивый старт абзаца.

Правило:

strictBottomHyphenation = true
11. Не допускать “одно слово + маленький зазор до края страницы”

Даже если последняя строка формально проходит по высоте, нельзя оставлять её, если это:

одно слово

короткий хвост

строка, после которой до границы страницы остаётся совсем маленький технический зазор.

Такой случай нужно считать типографически плохим break point.

12. Финальная проверка страницы перед commit

После построения страницы движок обязан выполнить post-layout validation:

validatePage():
    noClippedLines
    noFooterCollision
    noSingleWordBottomLine
    noTooShortBottomLine
    noMissingHyphenOnSplitWord
    hyphenCountWithinLimit
    noEarlyBreakWithUnnecessaryWhitespace
    noUglyParagraphSplit

Если хотя бы одно условие нарушено — страница пересобирается.

13. Приоритеты правил

Layout engine должен применять правила в таком порядке:

Критические

no clipped lines

no text in footer safe zone

no missing hyphen on split word

Очень важные

no single-word bottom line

no too-short bottom line

no ugly paragraph split

Важные

keep hyphen count low

minimize whitespace

maximize fill without breaking higher-priority rules

То есть нельзя “заполнить страницу красивее”, если для этого ломается footer safe zone или появляется тупой хвост.

14. Разделить три разных понятия

Движок должен чётко различать:

Physical fit — физически влезает ли строка

Typographic fit — хорошо ли выглядит этот break point

Optimal fit — лучший ли это вариант среди доступных.

Сейчас система, похоже, проверяет только первое. Нужно внедрить второе и третье.

15. Итоговое поведение

После исправления система должна гарантировать:

внизу страницы не остаётся одно слово или короткий хвост

последняя строка не визуально не наползает на зону нумерации

перенос слова всегда отображается с дефисом

переносов слов немного и они не идут грязной серией

страница не переносится слишком рано, если ещё можно корректно добавить текст

page break выбирается по качеству, а не только по высоте.
