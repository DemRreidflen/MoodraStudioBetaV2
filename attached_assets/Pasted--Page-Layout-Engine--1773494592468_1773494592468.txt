Задача: удалить старый рендерер страниц и реализовать новый Page Layout Engine, который верстает страницы как настоящие издательства (InDesign / Kindle), соблюдая правила типографики: отсутствие висячих строк, корректные переносы, отсутствие обрезаний, фиксированная зона footer, локализация подписей, адекватная верстка технических страниц (титул, копирайт, содержание).

1) Общая архитектура (обязательная)

Удалить/деактивировать существующий модуль рендера страниц.

Ввести архитектуру:

Text Engine -> Pagination Engine -> Page Renderer -> Export (PDF/HTML)

Page model: каждая страница = Header | ContentArea | Footer (отдельные независимые слои).

2) Структура страницы (жёстко и фиксировано)

Footer: фиксированная высота в коде (например footerHeight = 70px в абсолютных единицах реальной страницы / mm при генерации PDF). Пользователь не может менять расстояние от текста до footer и положение footer. Удалить любые UI-параметры, которые меняют это.

Header: фиксированная высота.

ContentArea: вычисляется строго по формуле:

contentHeight = pageHeight - headerHeight - footerHeight - pageTopMargin - pageBottomMargin

Никакой overflow:hidden у страницы, который обрезает текст. Контент рендерится только в ContentArea.

3) Pagination engine — алгоритм (минимум для реализации)

Разбить документ на параграфы.

Для каждого параграфа разбить на слова и вычислить возможные переносы (hyphenation points) через hyphenation engine.

Составлять строки по ширине (line box) с учётом текущего размера шрифта, межстрочного интервала и отступов.

Добавлять по одной строке в ContentArea; перед добавлением проверять:

если нижняя граница следующей строки > contentAreaBottom — НЕ добавлять строку, начать новую страницу.

проверка widow/orphan (см. пункт 4) — при необходимости переместить блок.

проверка «плохих концов строк» (см. пункт 5).

При любых изменениях параметров верстки (шрифт, leading, поля, колонки) — полный пересчёт pagination для всего документа.

Пример псевдо:

for paragraph in document:
  lines = breakParagraphToLines(paragraph)
  for line in lines:
    if line.bottom > contentArea.bottom:
      startNewPage()
    placeLine(line)
4) Widow / Orphan control (обязательно)

Нельзя оставлять 1 строку абзаца внизу страницы (widow) и 1 строку вверху (orphan).

Правило: минимум minKeepLines = 2.

Логика:

if remainingLinesOnPage < minKeepLines:
  moveEntireParagraphToNextPage()
else if paragraphWouldStartWithSingleLineOnNextPage:
  moveOneLineUpOrDownToAvoidSingleLine()
5) Контроль концов строк (типографика)

Запретить строки, заканчивающиеся одиночной запятой, коротким предлогом, союзом или одиночным тире.

Если строка заканчивается «плохим» токеном → попытка переноса предыдущего слова на следующую строку.

Если это не решает проблему — перебалансировать соседние строки или переместить часть абзаца на следующую страницу (см. paragraph balancing).

6) Hyphenation (переносы)

Интегрировать hyphenation engine (рекомендация: Hyphenopoly.js для браузерного рендера или серверный словарный движок для PDF).

Правила:

Не переносить слова < 5 букв.

Не переносить имена и аббревиатуры.

Не более maxHyphensPerPage = 3.

Не более maxConsecutiveHyphenLines = 2.

Переносы — крайняя мера; предпочтение — перераспределению слов/строк.

7) Optical margin alignment (висячая пунктуация)

Поддержать оптическое смещение знаков препинания у левых/правых краёв (minor overhang) для более ровного визуального края текста.

8) Paragraph balancing (визуальный баланс)

Если последняя строка абзаца слишком короткая (1–2 слова), алгоритм пытаетcя:

перераспределить слова между линиями

при возможности применить один перенос слова

в крайнем случае — переместить часть абзаца на следующую страницу

Цель — равномерные линии по длине и отсутствие «косых» разрывов.

9) Технические страницы (титул, копирайт, содержание)

Титульная страница: без аннотации. Только Title, Subtitle, Author, визуальные блоки.

Страница копирайта:

Структура (сверху вниз): права (copyright text) → большой отступ → заголовок «Аннотация» (локализуется) → аннотация (ограничена по объёму) → нижний якорный блок (информация о книге), который всегда закреплён у нижнего края страницы (редактор, дизайн обложки, правообладатель, год). Нижний блок фиксирован относительно нижнего края страницы; расстояние до края — фиксированное и неизменяемое.

Аннотация: заголовок локализуется под язык книги; сама аннотация ограничена по количеству символов/строк, соответствующих area.

Нельзя позволять аннотации «выталкивать» нижний блок за границы — если аннотация превышает доступный объём, показывать предупреждение/require overflow → перенести остаток аннотации на след. страницу.

10) Локализация / i18n

Все подписи: «Аннотация», «Правообладатель», «Редактор» и т.д. — берутся из таблицы локалей (i18n).

Реализовать поддержку минимум: en/ru/de (и легко расширяемо).

Формат: ICU / JSON локаль.

11) UI / настройки пользователя

В редакторе (Notion-style): запретить выбор размера шрифта и leading — редактор только для контента.

Все типографические параметры (font size, leading, margins, column width) настраиваются только в модуле верстки (layout settings) и при изменении запускают полный пересчёт pagination.

Удалить возможность менять расстояние от текста до footer.

12) Техническая реализация (рекомендации)

Не рендерить страницы как один div с overflow; использовать:

либо canvas/SVG серверный PDF рендер (PDFKit, Puppeteer + precise CSS),

либо виртуизированный renderer, который рисует страницы по вычисленным line boxes.

Hyphenation: Hyphenopoly.js (браузер) или серверный словарь (для PDF).

Локализация: ICU MessageFormat / i18n JSON.

Тесты: unit tests для pagination (edge cases), e2e для вёрстки (генерация PDF → ручная проверка), автоматические проверки на widows/orphans и на отсутствие обрезанных строк.

13) Acceptance Criteria (чёткие чек-поинты)

Генерируемая страница НЕ содержит ни одной обрезанной строки (ни одной).

Номер страницы занимает только footer, текст не перекрывает footer.

Нет висячих строк: ни 1 строки абзаца внизу или вверху страницы.

Страница копирайта: нижний блок всегда у нижнего края, аннотация ограничена и локализована.

При изменении размера шрифта/leading/полей — полный пересчёт pagination + корректный результат.

Переносы слова работают по словарным правилам и не превышают заданных лимитов.

Локализация подписей работает корректно для текущей языковой версии книги.

14) Тестовый набор (что прогонять после внедрения)

Документ с абзацами разной длины, длинными словами, спецсимволами — проверить отсутствие обрезаний.

Набор абзацев, где последний абзац страницы может породить widows/orphans — проверить контроль.

Аннотация, длиной на 120% от доступной области — проверить её ограничение и перенос на следующую страницу.

Изменение шрифта/leading в настройках верстки — проверить полный рефлоу.

Проверка локализации (ru/en/de) на странице копирайта.

15) Сообщение команде/AI (коротко и строго)

Удалить текущий layout module. Реализовать новый Pagination Engine по описанным правилам. Это критично: не патчить старый код — заменить/пересобрать. Убедиться, что footer — фиксирован и неизменяем. Подготовить unit и visual tests.
